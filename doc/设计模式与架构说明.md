# 硫化排程系统 - 设计模式与架构说明

## 一、设计模式应用详解

### 1. 策略模式 (Strategy Pattern)

**应用场景**：排程算法的可扩展实现

**代码位置**：
- `com.zlt.aps.lh.strategy.ScheduleStrategy` - 策略接口
- `com.zlt.aps.lh.strategy.AbstractScheduleStrategy` - 抽象策略基类
- `com.zlt.aps.lh.strategy.impl.BalancedScheduleStrategy` - 均衡策略实现

**设计意图**：
将排程算法封装成独立的策略类，使得排程算法可以独立于客户端变化，支持运行时动态切换不同的排程策略。

**类结构**：
```
┌─────────────────────────┐
│   <<interface>>         │
│   ScheduleStrategy      │
├─────────────────────────┤
│ + getName()             │
│ + schedule()            │
│ + selectBestMachine()   │
│ + calculateScheduleQty()│
└───────────┬─────────────┘
            │ implements
┌───────────┴─────────────┐
│ <<abstract>>            │
│ AbstractScheduleStrategy│
├─────────────────────────┤
│ # validatorChain        │
│ # capacityValidator     │
│ # mouldValidator        │
├─────────────────────────┤
│ + scheduleProduct()     │
│ + generateScheduleResult│
└───────────┬─────────────┘
            │ extends
    ┌───────┴───────┐
    │               │
┌───┴────┐    ┌────┴───┐
│Balanced│    │Priority│ (可扩展)
│Strategy│    │Strategy│
└────────┘    └────────┘
```

**应用代码**：
```java
// 策略接口定义
public interface ScheduleStrategy {
    String getName();
    void schedule(ScheduleContext context);
    LhMachineInfo selectBestMachine(ScheduleContext context, MonthPlanProd prod,
                                     List<LhMachineInfo> availableMachines,
                                     Date scheduleDate, String shift);
}

// 具体策略实现 - 均衡策略
@Component
public class BalancedScheduleStrategy extends AbstractScheduleStrategy {
    @Override
    public LhMachineInfo selectBestMachine(...) {
        // 选择已排数量最少的机台（均衡负载）
        return availableMachines.stream()
            .min(Comparator.comparingInt(m -> 
                context.getMachineDaySchedule(m.getMachineCode(), scheduleDate)
                       .getShiftScheduledQuantity(shift)))
            .orElse(null);
    }
}
```

### 2. 工厂模式 (Factory Pattern)

**应用场景**：策略对象的创建和管理

**代码位置**：
- `com.zlt.aps.lh.strategy.ScheduleStrategyFactory`

**设计意图**：
封装策略对象的创建逻辑，客户端无需知道具体策略类的创建细节，实现解耦。

**类结构**：
```
┌─────────────────────────────┐
│   ScheduleStrategyFactory   │
├─────────────────────────────┤
│ - strategyMap: Map<String,  │
│               ScheduleStrat-│
│               egy>          │
├─────────────────────────────┤
│ + getStrategy(type):        │
│       ScheduleStrategy      │
│ + registerStrategy()        │
│ + getAllStrategyTypes()     │
└─────────────────────────────┘
```

**应用代码**：
```java
@Component
public class ScheduleStrategyFactory {
    @Autowired
    private BalancedScheduleStrategy balancedScheduleStrategy;
    
    private Map<String, ScheduleStrategy> strategyMap = new HashMap<>();

    @PostConstruct
    public void init() {
        // 注册策略
        strategyMap.put(StrategyTypeEnum.BALANCED.getCode(), balancedScheduleStrategy);
        // 可扩展注册其他策略
    }

    public ScheduleStrategy getStrategy(String strategyType) {
        ScheduleStrategy strategy = strategyMap.get(strategyType);
        if (strategy == null) {
            strategy = balancedScheduleStrategy; // 默认策略
        }
        return strategy;
    }
}
```

### 3. 责任链模式 (Chain of Responsibility)

**应用场景**：约束校验的组合执行

**代码位置**：
- `com.zlt.aps.lh.validator.ConstraintValidator` - 校验器接口
- `com.zlt.aps.lh.validator.ValidatorChain` - 校验器链
- `com.zlt.aps.lh.validator.impl.*` - 具体校验器实现

**设计意图**：
将多个校验器串联成链，请求沿着链传递，直到有校验器处理失败或全部通过。支持动态添加/移除校验器。

**类结构**：
```
┌─────────────────────────┐
│   <<interface>>         │
│   ConstraintValidator   │
├─────────────────────────┤
│ + validate()            │
│ + getAvailableMachines()│
│ + getName()             │
└───────────┬─────────────┘
            │
    ┌───────┴───────┬───────────────┬───────────────┐
    │               │               │               │
┌───┴──────┐  ┌─────┴────┐  ┌───────┴────┐  ┌──────┴─────┐
│Validator │  │Machine   │  │Specify     │  │Capacity    │
│Chain     │  │Availabil-│  │Machine     │  │Validator   │
│          │  │ityValid. │  │Validator   │  │            │
└──────────┘  └──────────┘  └────────────┘  └────────────┘
```

**应用代码**：
```java
@Component
public class ValidatorChain implements ConstraintValidator {
    private List<ConstraintValidator> validators = new ArrayList<>();

    @Override
    public ValidationResult validate(ScheduleContext context, MonthPlanProd prod, 
                                      LhMachineInfo machine, Date scheduleDate, String shift) {
        // 责任链：依次执行校验，任一失败则终止
        for (ConstraintValidator validator : validators) {
            ValidationResult result = validator.validate(context, prod, machine, scheduleDate, shift);
            if (!result.isValid()) {
                return result; // 校验失败，终止链
            }
        }
        return ValidationResult.success();
    }

    @Override
    public List<LhMachineInfo> getAvailableMachines(...) {
        // 责任链：依次过滤，逐步缩小范围
        List<LhMachineInfo> availableMachines = allMachines;
        for (ConstraintValidator validator : validators) {
            availableMachines = validator.getAvailableMachines(...);
            if (availableMachines.isEmpty()) {
                break; // 无可用机台，终止链
            }
        }
        return availableMachines;
    }
}
```

**校验器执行顺序**：
```
请求 → 机台可用性校验 → 定点机台校验 → 产能校验 → 模具可用性校验 → 结果
         ↓                  ↓              ↓              ↓
     停机/保养检查      定点限制检查    产能剩余检查    模具状态检查
```

### 4. 模板方法模式 (Template Method Pattern)

**应用场景**：排程策略的通用流程

**代码位置**：
- `com.zlt.aps.lh.strategy.AbstractScheduleStrategy`

**设计意图**：
定义排程算法的骨架，将可变部分延迟到子类实现。子类可以重写特定步骤，但不改变算法结构。

**类结构**：
```
┌─────────────────────────────────────┐
│  <<abstract>>                       │
│  AbstractScheduleStrategy           │
├─────────────────────────────────────┤
│ # validatorChain                    │
│ # capacityValidator                 │
│ # mouldValidator                    │
├─────────────────────────────────────┤
│ + scheduleProduct()      // 模板方法 │
│ + calculateScheduleQuantity()       │
│ + generateScheduleResult()          │
│ # isWorkDay()                       │
│ # getShiftList()                    │
│                                     │
│ <<abstract>>                        │
│ + schedule()             // 子类实现 │
│ + selectBestMachine()    // 子类实现 │
└─────────────────────────────────────┘
```

**应用代码**：
```java
public abstract class AbstractScheduleStrategy implements ScheduleStrategy {

    // 模板方法：定义排程产品的骨架流程
    @Override
    public List<LhScheduleResult> scheduleProduct(ScheduleContext context, MonthPlanProd prod,
                                                   Date scheduleDate, String shift) {
        List<LhScheduleResult> results = new ArrayList<>();

        // 步骤1：获取剩余需求数量
        int remainingQuantity = context.getRemainingQuantity(prod.getId());
        if (remainingQuantity <= 0) {
            return results;
        }

        // 步骤2：获取可用机台列表（通过校验器链）
        List<LhMachineInfo> availableMachines = validatorChain.getAvailableMachines(
                context, prod, scheduleDate, shift);
        if (availableMachines.isEmpty()) {
            return results;
        }

        // 步骤3：选择最优机台（子类实现）
        LhMachineInfo bestMachine = selectBestMachine(context, prod, availableMachines, scheduleDate, shift);

        // 步骤4：计算排程数量
        int scheduleQuantity = calculateScheduleQuantity(context, prod, bestMachine, scheduleDate, shift);

        // 步骤5：生成排程结果
        LhScheduleResult result = generateScheduleResult(context, prod, bestMachine, scheduleDate, shift, scheduleQuantity);
        if (result != null) {
            results.add(result);
        }

        return results;
    }

    // 抽象方法：由子类实现具体的选择逻辑
    @Override
    public abstract LhMachineInfo selectBestMachine(ScheduleContext context, MonthPlanProd prod,
                                                     List<LhMachineInfo> availableMachines,
                                                     Date scheduleDate, String shift);
}
```

## 五、设计模式优势总结

| 设计模式 | 解决的问题 | 在本项目中的优势 |
|---------|-----------|-----------------|
| 策略模式 | 排程算法可变 | 支持多种排程策略，易于扩展新策略，策略可运行时切换 |
| 工厂模式 | 策略创建复杂 | 封装创建逻辑，客户端无需知道具体实现类 |
| 责任链模式 | 多重校验组合 | 校验器可自由组合，支持动态增减，单一职责 |
| 模板方法模式 | 流程复用 | 统一排程骨架，子类只需实现差异化逻辑 |
| 构建者模式 | 复杂对象构建 | 分步构建上下文，代码更清晰 |
